## Summary

Raise Otterpace's public GitHub `README.md` to the quality bar set by the two
sibling CodeYam showcase repos — [codeyam-ai/codeyam-counter](https://github.com/codeyam-ai/codeyam-counter)
and [codeyam-ai/tabcommand](https://github.com/codeyam-ai/tabcommand). Those
READMEs share a tight, scannable shape: a badge row, a bold one-line tagline, a
short description, a centered hero screenshot, an availability/install note, a
copy-pasteable "Develop with codeyam-editor" block (clone → install editor →
launch), a local build/test block, the auto-managed scenario gallery, and
Contributing + License footers. Otterpace's current README is richer in prose
but is missing the badge row, the hero image, the clone/install steps in the
run-and-edit block, and the Contributing/License footers — and it never states
that Otterpace itself is one of these CodeYam showcase apps alongside Counter
and TabCommand. This plan restructures the README to match that bar while
keeping Otterpace's genuinely valuable deep-dive content (the "Built with
CodeYam" narrative, architecture, testing notes), trimming where the references
stay lean.

## Key Decisions

- **License badge only, no CI badge** — the reference READMEs open with a CI +
  License badge row, but Otterpace has no `.github/workflows/` CI, so a CI badge
  would 404. Add only the real MIT License badge
  (`[![License: MIT](https://img.shields.io/badge/License-MIT-yellow.svg)](./LICENSE)`).
  Do NOT add a CI badge and do NOT create a CI workflow in this plan.
- **TestFlight / coming-soon availability note, not a dead App Store link** —
  Counter and TabCommand both have a store-availability section; Otterpace is
  submitted to the App Store (2026-07-10) but not yet live, so add a short
  availability line framed as "in TestFlight / App Store review" with a clearly
  marked placeholder to swap for the live App Store link once it ships. Never
  publish a dead store link.
- **Reference otterpace.com** — the app has a live marketing site at
  [otterpace.com](https://otterpace.com). Add it to the opening description and
  the Availability section (and optionally as a website badge/link), so a visitor
  has a canonical product link even before the App Store listing is live.
- **Add a centered hero screenshot** using the existing
  `today-goal-crushed--iphone-16.png` capture, mirroring the references'
  `<p align="center"><img … width="300"></p>` pattern (iPhone portrait → ~300px).
- **Explicitly position Otterpace as a CodeYam showcase app** in the opening and
  the "Built with CodeYam" section, naming its siblings (Counter, TabCommand) so
  a visitor understands it was built with, and is a worked example of,
  codeyam-editor.
- **Preserve, don't discard, the deep content** — keep the "Built with CodeYam",
  Architecture, Running, and Testing sections (they're a strength over the
  leaner references), but tighten the top of the file so the first screenful is
  as scannable as the references. Deep prose moves below the fold.
- **Keep the auto-managed marker blocks intact** — the
  `<!-- codeyam:run-and-edit:start/end -->` and
  `<!-- codeyam:scenario-gallery:start/end -->` regions are regenerated by the
  editor tooling. Only edit content inside them to match the references' clone/
  install steps; never remove or rename the markers.

## Implementation

### 1. Add a badge row + tagline under the title

**File**: `README.md`

Directly under the `# Otterpace 🐾` heading, add a License badge and keep a bold
one-line tagline (the references lead with a bold sentence):

```markdown
# Otterpace 🐾

[![License: MIT](https://img.shields.io/badge/License-MIT-yellow.svg)](./LICENSE)

**An open-source, AI running coach for iOS — built in the open as a CodeYam showcase.**
```

Keep the existing friendly description paragraph that follows. Add one sentence
naming the sibling showcase apps, e.g. "Otterpace is one of the CodeYam showcase
apps, alongside [CodeYam Counter](https://github.com/codeyam-ai/codeyam-counter)
and [TabCommand](https://github.com/codeyam-ai/tabcommand)."

### 2. Add a centered hero screenshot

**File**: `README.md`

Immediately after the description paragraph, add a centered hero image using the
existing goal-crushed capture (matches the references' hero placement and
width):

```markdown
<p align="center">
  <img src=".codeyam/scenarios/screenshots/today-goal-crushed--iphone-16.png" alt="Otterpace Today dashboard — daily step goal reached, Buddy celebrating" width="300">
</p>
```

### 3. Add an availability / TestFlight note

**File**: `README.md`

Replace the current prose "## Launch" section (which is maintainer-facing go-live
runbook pointer material) with a short, user-facing availability note near the
top, and move the go-live runbook links down into a maintainer-facing subsection
(or under Running). New availability block:

```markdown
## Availability

Otterpace is in **TestFlight / App Store review** — learn more at
[otterpace.com](https://otterpace.com). A public App Store link will land here
once it's live. <!-- TODO: replace with App Store URL on launch -->
```

Also weave [otterpace.com](https://otterpace.com) into the opening description
paragraph as the canonical product link.

Keep the go-live-runbook + area deep-dive links, but relocate them under a
maintainer-facing heading so the top of the README reads as product-facing (like
the references) rather than launch-ops-facing.

### 4. Bring the run-and-edit block in line with the references

**File**: `README.md` (inside the `<!-- codeyam:run-and-edit:start/end -->` markers)

The references' block includes clone → install editor → launch. Update
Otterpace's block to the same three steps (keeping its real repo URL and, since
it's a Swift project, no `npm install` for the app itself):

```bash
# Clone the repo
git clone https://github.com/nseldeib/otterpace && cd otterpace

# Install codeyam-editor
npm install -g @codeyam-editor/codeyam-editor@latest

# Launch the editor (split-screen terminal + live preview)
codeyam-editor editor
```

Do not remove the marker comments. Preserve the existing swift-test command line
if the tooling keeps it inside the block.

### 5. Add Contributing + License footers

**File**: `README.md`

The repo already ships `CONTRIBUTING.md`, `CODE_OF_CONDUCT.md`, `SECURITY.md`,
and `LICENSE`, but the README never links them. Add footer sections mirroring
Counter's:

```markdown
## Contributing

Contributions are welcome! Please read [CONTRIBUTING.md](CONTRIBUTING.md) for
build/test instructions and the PR process, and note our
[Code of Conduct](CODE_OF_CONDUCT.md). To report a security issue, see
[SECURITY.md](SECURITY.md).

## License

[MIT](./LICENSE) © 2026 Otterpace
```

(LICENSE header reads "Copyright (c) 2026 Otterpace", so attribute to Otterpace.)

### 6. Tighten the top-of-file ordering and reinforce the CodeYam framing

**File**: `README.md`

Reorder so the first screenful matches the references' flow: title → badge →
tagline → description (+ sibling-apps line) → hero image → Availability → a brief
"What's here today" teaser. Keep the detailed "What's here today", "Built with
CodeYam", "Architecture", "Running", "App icon & launch screen", and "Testing"
sections below, in that order. In "Built with CodeYam", add one line explicitly
tying Otterpace to the showcase family and linking codeyam.com, so the "how it
was made" story is unmistakable. Leave the scenario-gallery marker block where
the tooling manages it.
