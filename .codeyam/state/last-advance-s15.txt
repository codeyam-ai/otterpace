For each component with an isolation route, register ONE scenario per state worth
showing — walk the state-axis checklist in the `isolation-routes` step (Default /
Edge data / Boundary values / Visual states / All-of-one-kind). A single Default
scenario per component leaves it under-covered; the audit treats a `componentName
== entry.name` match as the satisfier, so you get credit for every distinct state.

Demonstrating every meaningful state is the core value of codeyam — not a checkbox.
A scenario that seeds data but renders blank, or shows only one trivial state, is a
BUG: seed it richer and recapture. Reserve `n-a` for states that genuinely cannot
occur on this view, with a reason that says why — never for a state you skipped.

ANTI-PATTERN — the empty-shell seed. A scenario whose array/list props are all `[]`
and objects all `null`/placeholder renders an empty shell — a bug, not a state.
Default MUST populate every list with realistic rows, fill optional fields, and use
counts that exercise pluralization, so it looks like the component doing its job.
Keep empty / loading / error as SEPARATE, clearly-named scenarios; a genuinely-empty
capture is only legitimate when the empty state is the point.

Components without an isolation route MUST carry an `untestabilityReason` on their
glossary entry — and only the GENUINE wrapper shape (`PageShell` / `BrandMark` /
`CategoryTag`: one element, no branching, no iteration, ≤1 meaningful prop) is
accepted as `trivial-wrapper`. `glossary-add` reads the source and rejects the kind
when the component shows real render complexity; the audit mirrors the check so
mis-classified entries surface in `visual-components-without-coverage`. The
write-time check also rejects a page-level application scenario as a visual-component
satisfier — page coverage is NOT a component substitute.

Once each scenario is registered + captured here, its EMITTED slug (in the
`register` response) is the legal value to cite on the matching `glossary-add` entry
at the `register-incremental` step — that's when the visual-component coverage
contract becomes satisfiable. The slug is `slugify(scenario.name)`; cite the printed
slug, never a guess.

[ ] Register a scenario for every visual component (bulk; one per state worth showing)
    Canonical shape — `componentName` + isolation-route `url` + (optional)
    `mocks.http` for any internal API calls:

```json
{
  "name": "ArticleCard - Default",
  "componentName": "ArticleCard",
  "url": "/isolated-components/ArticleCard?s=Default",
  "dimensions": ["Desktop"],
  "mocks": {
    "http": { "GET /api/article/1": {"status": 200, "body": {"id": 1, "title": "Iced matcha"}} }
  }
}
```

    Before hand-writing an HTTP mock body, start from an observed response:
    `codeyam-editor-dev editor catalog-export --route /api/article` (falls back to the
    persisted catalog when the server is down). Where `.codeyam/mockable/<app>.json`
    declares a `responseSchema`, `register` validates the body and bails loud on a
    mismatch; else an observed-sample divergence is a non-fatal warning.

    Bulk-register: write the array to a per-invocation scratch path (unique
    filename per call) and run `codeyam-editor-dev editor register
    @.codeyam/tmp/register-<batch-tag>.json`. The file auto-deletes on success — do
    NOT reuse a shared path. URL MUST be an isolation route; omit `mocks` when the
    component has no internal API calls. With `components.registryFile` set in
    `stack.json`, register validates every `componentName` and fails loud on unknowns.

    Streaming / transcript mocks (same `mocks` object as `http`). A WebSocket/SSE
    feed uses `mocks.streams` keyed by an HTTP-style endpoint; a terminal/agent
    view uses `mocks.transcripts` keyed by the PTY endpoint (`"/ws/terminal"`). The
    value is a timed frame list: a transcript frame is either `bytes` (terminal
    output, ANSI included) OR `event` (a lifecycle message like `agent-idle`), never
    both; a stream frame is `{at, kind, data}`. `onEnd` is `close` (default) | `idle`
    | `loop`. **Every endpoint AND every `event` name must be declared in
    `.codeyam/mockable/<app>.json`** — `register` rejects an undeclared endpoint or
    misspelled event. Capture a real state with `record-transcript --scenario <slug>`.

    Interaction scenarios (the `interactions` field). A resting-render screenshot
    shows nothing about what a component does when touched — a hover action bar, an
    expanded section, an open modal, a validation error all stay invisible. If a
    component binds event handlers and has behavior worth showing, register a
    scenario whose `interactions` list drives it, so the capture shows the
    INTERACTED state. Each step is `click` / `fill` / `press` / `hover`, with a
    `text` (visible label, preferred) or CSS `selector` target and optional `value`
    (text to `fill`, or the key for `press`). The sequence replays before every
    capture and `recapture-stale`, so the state stays fresh. A target matching
    nothing fails the capture loud (with candidate labels), never silently.

```json
{
  "name": "UrlRow - Hover Actions Revealed",
  "componentName": "UrlRow",
  "url": "/isolated-components/UrlRow?s=Default",
  "dimensions": ["Desktop"],
  "interactions": [
    { "action": "hover", "text": "https://example.com/articles" }
  ]
}
```

    Only add `interactions` for components that GENUINELY have interaction-only
    state. A purely presentational component (tag, badge, static card) has nothing
    to reveal — a Default scenario is complete; an interaction just re-captures it.

Non-Serializable Props Reference (use these `__type` markers in scenario props):

  | `__type`      | JSON                                                          | Runtime value           |
  |---------------|---------------------------------------------------------------|-------------------------|
  | `"noop"`      | `{"__type": "noop"}`                                          | `() => {}`              |
  | `"undefined"` | `{"__type": "undefined"}`                                     | `undefined`             |
  | `"Ref"`       | `{"__type": "Ref"}`                                           | `{ current: null }`     |
  | `"Date"`      | `{"__type": "Date", "value": "2024-01-15T10:30:00Z"}`         | `new Date(value)`       |
  | `"Set"`       | `{"__type": "Set", "values": [1, 2, 3]}`                      | `new Set([1, 2, 3])`    |
  | `"Map"`       | `{"__type": "Map", "entries": {"k": "v"}}`                    | `new Map([["k","v"]])`  |
  | `"ReactNode"` | `{"__type": "ReactNode", "text": "Hello"}`                    | `"Hello"`               |
  | `"ReactNode"` | `{"__type": "ReactNode", "element": "div", "children": "…"}`  | `createElement("div")`  |

  Required-callback props: always `{"__type": "noop"}`, never `null`.
