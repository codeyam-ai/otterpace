Sync local state with the remote so the next commit lands on a tree
that already includes any upstream work. The command runs once,
end-to-end, with no agent loop:

1. `git fetch` the upstream.
2. If the remote is ahead, `git pull --rebase`. The custom CodeYam
   merge drivers (verified BEFORE the rebase) merge state files
   cleanly; source-file conflicts surface as a hard `RESOLVE NOW:`
   bail.
3. After a successful rebase, the inline auto-recovery sequence runs:
   `reconcile-registry --auto-apply --allow-removes` ‚Üí
   `reconcile-glossary --auto-apply` ‚Üí `analyze-imports` ‚Üí
   `verify-test-cache --partial` (informational). Any failure surfaces
   as a structured `recoveryError: { step, message, suggestedAction }`
   in the JSON output ‚Äî no agent loop, no second command.

Checklist:
[ ] Run `codeyam-editor-dev editor pre-commit-sync`.

    Running `pre-commit-sync` ALWAYS blocks until this session reaches the
    head of the commit queue ‚Äî there is no fleet-vs-solo judgment call to
    make, and no flag to add. On a deep queue this can take a while; that
    is EXPECTED, not a hang.
    - Run it EXACTLY ONCE and let it block. Do NOT poll `branch-queue
      status` in a loop, do NOT re-run it repeatedly, and do NOT ask the
      operator whether to wait or skip.
    - The periodic `Commit queue: still waiting ‚Äî position N of M ‚Ä¶` lines

    - An abandoned head ahead of you (a crashed or wedged session) is
      reclaimed AUTOMATICALLY by the lease sweep ‚Äî you do not need to do

      do NOT pipe it into a background task, do NOT wrap it so you can keep
      working while it runs. Being blocked IS the mechanism ‚Äî the moment

      what leads to the confused "queue looks empty, go commit manually"
      misread.

    `--recover` runs `git pull --rebase --autostash` ‚Üí

    NOTE on the underlying mechanics: `git stash` / `git pull --rebase

    performs git's default three-way merge, which ignores `.gitattributes`
    `merge=` assignments, so union state files (glossary, test registry,
    dependency graph) carried through an autostash silently resolve to upstream
    and your session's adds are dropped. That is exactly why `--recover` (and
    the hand recovery) run `post-merge-drift-sweep` after the autostash rebase
    to re-apply that union state before re-running `pre-commit-sync`.

    The duplicate-slug case is the post-rebase shape where the queue copy was
    already deleted in the worktree but the deletion is unstaged.
    `plan-cleanup-duplicates` now stages that deletion for you ‚Äî you do NOT
    need to `git add` the deleted queue copy by hand.
