Always run tests through the configured runner command via `refresh-tests`

The audit is a HARD GATE; the next slug will refuse to advance until the

[ ] Run `codeyam-editor-dev editor audit --summary-only`
    Bounded ≤30-line summary; never `tail`/`head` full output. See CLAUDE.md `## Audit output discipline` for `--only <ID>` / `--format json` drill-ins.
    If the audit bails reporting cache staleness, that is the only place
    in the workflow where a flag-free `codeyam-editor-dev editor refresh-tests`
    may run before `finalize`. Run it, then re-run `audit`. If the audit

    `codeyam-editor-dev editor refresh-tests --test <name>` to update just
    that entry's cache. NEVER run `refresh-tests --force` to update the
    cache for a single test — `--force` runs the entire suite, is
    extremely slow, and is reserved for the `finalize` step (where
    `session-finalize` runs it automatically). Other targeted flags:
    `--changed` (run only what changed since HEAD), `--runner <name>`

    cue to drop back to a targeted flag.

    registry against the test cache, so it only needs the partition(s)
    you touched this branch to be fresh — NOT the whole suite. When
    `reconcile-registry` bails on a partial or stale/aborted cache, read
    the bail: it now prints the exact, copy-pasteable
    `refresh-tests --partition <touched>` command that fixes it. Run that
    command verbatim, then re-run `reconcile-registry`. Reach for
    `--force` / a full-suite refresh ONLY if a partition refresh still
    leaves the cache incomplete — escalating straight to `--force` is the
    redundant full-suite churn this contract exists to prevent.

    Environmental drift is not a real failure — defer it, don't `--force`

    full-suite `refresh-tests --force` to clear them — that is ~20+ minutes
    for zero signal. Carry them to `session-finalize`, which runs the full
    suite once before the branch merges.

    aborted mid-suite in a crate your change never touched,

[ ] Fix every drift the audit names — registry first, glossary second
    Registry / runner drift must be resolved before advancing. Use
    `register-test`, `remove-test`, `rename-test`, or `platform-gate`
    — there is no suppression primitive. If a test is intentionally
    not running on a given host, that is `platform_gate`. If a test
    is intentionally not running at all, that is `#[ignore]` (Rust)
    or `it.skip()` (vitest), not a registry-side marker.
    Reconcile in bulk via `codeyam-editor-dev editor reconcile-registry`
    (`--auto-apply` runs the unambiguous actions). What `--auto-apply`
    can and cannot do depends entirely on what metadata the runner
    emits for each test:
      * Runner emits file + line (jest, etc.): auto-applied directly.
      * Runner emits file but no line (vitest today): line is resolved
        from source by the runner-shape-dispatched resolver, and the
        add is auto-applied. No manual `register-test` needed.
      * Runner emits no file path (cargo today, and any future runner
        with the same output shape): the entry will ALWAYS land in
        `skipped` and `reconcile-registry` will exit with code 2.
        There is nothing for `--auto-apply` to anchor an add to.
    For the cargo case (which is the common one when you've just added
    a new `#[test]`), the printed skip line contains a complete,
    copy-pasteable `register-test` command with `--key`, `--runner`,
    and a placeholder `--file` / `--description` you fill in. Run that
    command for each skipped entry. The `--line` flag is optional for
    `register-test` — the source-resolver will fill it in.
    Do NOT re-run `reconcile-registry --auto-apply` expecting a different
    result for these entries — they will land in `skipped` again with
    the same message. Do NOT `git checkout .codeyam/test-registry.json`
    to "start fresh"; that destroys session work and does not change
    auto-apply's behaviour. The exit code is the contract: 0 = clean,
    2 = read the printed commands and run them.
    Removes are skipped by default; pass `--allow-removes` together with
    `--auto-apply` when you know the deletions are intentional (e.g. you
    just refactored tests away) to skip the per-key `remove-test`
    round-trips.
    For glossary integrity (`staleSources` / `staleTests`): remove the
    referenced entries from `.codeyam/glossary.json` (use

    incomplete entities by running targeted analysis on just the affected
    files. Do NOT run `codeyam-editor-dev editor analyze-imports` here — the
    audit runs targeted analysis automatically and is much faster.
