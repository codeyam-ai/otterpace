Every extracted visual component must end up satisfied by EITHER a
component scenario whose `componentName` matches OR an
`untestabilityReason`. Page-level (application) scenarios do NOT credit
a component entry — the audit matches by `componentName == entry.name`
or `pageFilePath == entry.file_path` (entry-point case only).
`glossary-add` enforces the same contract at write time.

Pipeline ordering — `isolate` → `register` (component scenario) →
`glossary-add` citing the EMITTED slug `register` prints (never a
guess). The glossary entry isn't registerable until `register-incremental`,
because `glossary-add` rejects a visual-component `scenarios` citation
before the slug exists in scenarios.json.

`codeyam-editor-dev editor isolate` resolves a component name from source
even when it's not yet in the glossary — the supported path for freshly
extracted components (run `analyze-imports` first when adding a new file
so the dep-graph picks it up). There is NO "register the entry, then
isolate" prerequisite; isolate-first skips the old `trivial-wrapper` bind.
Run it BEFORE hand-writing a scenario — let it tell you whether to capture
a visual scenario or fall back to a `testFile` satisfier. Never skip
`isolate` and hand-author an `/isolated-components/...` url: on a static
export (Astro / plain HTML) that url 404s on capture, and `register` now
rejects it loud rather than persisting an uncapturable ghost.

The `testFile` fallback is ONLY for stacks without an `isolate`
scaffolder. If `codeyam-editor-dev editor isolate <Component>` SUCCEEDS for
your stack (nextjs-prisma-sqlite, expo-react-native, any query-param
stack, …), you MUST capture a component scenario for every visual
component — a `testFile` (unit test) is NOT a visual scenario and does
NOT substitute. The coverage gate is isolation-aware: when your stack can
isolate, a component credited only by a `testFile` glossary entry is
surfaced as uncovered by `visual-components-without-coverage` and the
advance gate blocks. Only when `isolate` bails that your stack is
unsupported do you fall back — follow the recovery it prints: register the
component in the registry file named by `.codeyam/stack.json` →
`components.registryFile`, then satisfy the visual-component check with a
`testFile` entry. The Next.js scaffolding below applies to any stack whose
`isolate` succeeds — the route shape differs (expo-router
`app/isolated-components/<Name>.tsx`, Next.js `<Name>/page.tsx`) but the
`?s=` + capture-wrapper contract is the same.

    Verify: codeyam-editor-dev editor visual-components-without-coverage

    (a) ISOLATE + multi-state scenarios — the default for any component
        with a non-trivial prop surface (≥1 meaningful prop) OR more than
        one visual state. Walk the state-axis checklist (mirrors
        `app-scenarios-state-coverage`, trimmed for component level):
          - Default — typical props, the page's most common configuration.
          - Edge data — long / short / special-char / missing-optional-
            field. Exercises typography and null branches.
          - Boundary values — single vs many, min/max, zero counts.
            Exercises pluralization and empty/filled branches.
          - Visual states — loading / error / disabled / selected /
            featured / hover. One scenario per state worth showing.
          - All-of-one-kind — every row shares a normally-discriminating
            prop value (only when relevant).

        GOOD DATA IS NOT OPTIONAL. A scenario whose array/list props are all
        `[]` and whose objects are all `null`/placeholder renders an empty
        shell — that is a BUG to fix and recapture, not a state. The Default
        scenario must populate every list with realistic rows, fill optionals, and
        use counts that exercise pluralization — it must look like the
        component doing its job, not its empty state. Empty / loading /
        error belong in their OWN separate, clearly-named scenarios, never
        as the Default.

        Read the source AND its usage site to learn props/interface and
        container width. Create ONE isolation route per component with a
        scenarios map and ?s= query param (Next.js:
        app/isolated-components/ComponentName/page.tsx →
        /isolated-components/ComponentName). The route defines a
        `scenarios` object mapping names to props, reads `?s=Name`, and
        renders the component inside `id="codeyam-capture"` — the SAME
        shape the scaffolder emits:
          <div id="codeyam-capture">
            <Component {...props} />
          </div>

        CENTERING — leave it to the layout. The scaffolded
        `app/isolated-components/layout.tsx` already centers the capture
        (flex, `min-height:100vh`, centered). The page must NOT re-declare
        `min-h-screen`/`items-center`/`justify-center` or wrap the capture in
        `display:inline-block` — an inline-block shrink-to-fit with an inner
        `width:100%`/`minWidth` yields the off-center, arbitrary-width capture
        this exists to prevent. Don't fight the layout.

        WIDTH — match the component's REAL container, keyed to the Desktop
        capture viewport (1440×900). Add a thin width wrapper ONLY when the
        component needs a bounded width:
          - Full-width surface (header, nav, toolbar, banner, section, full
            pane): leave `#codeyam-capture` as-is, or wrap in
            `<div style={{ width:"100%" }}>` with NO `maxWidth`.
          - Fixed-width card / panel (grid cell, sidebar item): wrap in
            `<div style={{ width:"100%", maxWidth:<cell width> }}>` — the
            number comes from the real grid/column (card in a 3-col grid → ~384).
          - Intrinsic control (button, badge, chip): NO width wrapper — let
            it size to content.
        READ the width from the component's actual usage site, never a
        fabricated `minWidth`. If you can't find it, the component is
        full-width — use `width:"100%"`.

    (b) UNTESTABILITY REASON — for **genuine** trivial wrappers: no
        meaningful props, single visual state, layout-only (PageShell,
        BrandMark, SiteFooter) — one element, optionally forwarding
        `children`/`className`/`style`, no branching or iteration. Supply
        `untestabilityReason: {"kind":"trivial-wrapper","justification":"<one line>"}`.

        `glossary-add` rejects `trivial-wrapper` on a visual component
        whose source actually shows real render complexity (`.map(...)`
        over data, conditional JSX, more than ~3 distinct child
        elements, or more than one meaningful prop). When that
        rejection fires, the recovery is (a): run `editor isolate
        <Component>` and register multi-state scenarios — NOT to
        write a longer justification.

    NOT VALID: claiming a page-level (application) scenario as the sole
    satisfier for a component glossary entry. Page-level visual coverage
    is NOT a substitute — same way "Loaded with many" is not an Empty
    scenario. The write-time check in `glossary-add` enforces this. If
    you find yourself listing `scenarios: ["<page-app-scenario>"]` on a
    component entry whose `filePath` is NOT the page itself, use (a)/(b).

Isolation routes are committed to git — the layout guard blocks them in production.
