━━━ WAITING ON BACKGROUND WORK ━━━
A long editor command (`refresh-tests`, `session-finalize`, `rebuild-self`,
`recapture-stale`, `analyze-imports`) may end up running in the background —
either because the harness auto-backgrounded it, or because you launched a
genuinely long op with `run_in_background: true`. When that happens, TRUST
the harness completion notification; do NOT poll.

  - The harness re-invokes you automatically the moment the task exits — that
    re-invocation IS your completion signal. Do NOT spawn `sleep` /
    `until grep "CODEYAM_CMD_COMPLETE" … sleep` loops, do NOT re-read the
    task `.output` file on a timer, and do NOT `kill -0` spin. A bare `sleep`
    is blocked by the harness anyway, so the polling fallback never even
    works — it just burns turns.
  - If you need to block on the result inside the SAME turn, run the printed
    `codeyam-editor-dev editor wait-for <task-id>` line BARE, exactly once. It now
    BLOCKS to completion in one call and returns a single `WAIT_FOR COMPLETE`
    block — you do NOT re-issue it repeatedly. Re-issue ONLY if it reports
    `WAIT_FOR LIKELY_WEDGED` (a stale heartbeat — it names a `pid-status <pid>`
    command to confirm the process is dead before killing). Never hand-roll an
    `until grep … sleep` loop, and do NOT pipe it through `| tail`: the wrapped
    command self-reports a `CODEYAM_CMD_RUNNING` heartbeat even under a pipe, so
    piping only block-buffers and hides that liveness.
  - `Monitor` is the supported watch tool, but its schema must be loaded
    first or the call fails with `InputValidationError` and you fall back to
    the very polling this avoids. It is pre-loaded once at session start
    (`ToolSearch select:Monitor`); if it ever is not, load it before use.
  - Only if you genuinely need a hang-detector, schedule ONE long fallback
    (`ScheduleWakeup` ≥ 1200s) — never a short polling cadence. A fixed short
    interval just fires stale after the task already finished, forcing a
    context refetch to dismiss it.
  - TIE that fallback to the current step so a late firing self-dismisses:
    after you create the ScheduleWakeup, run
    `codeyam-editor-dev editor wakeup-arm --schedule-id <the ScheduleWakeup id you
    just created> --task <the backgrounded task id>`, embed the token it prints
    in the wakeup's prompt, and have the prompt instruct future-you to run
    `codeyam-editor-dev editor wakeup-check --token <token>` FIRST. If it reports
    STALE (the workflow advanced past the armed step, a different feature is
    active, or the session completed/reset), dismiss the wakeup WITHOUT
    re-fetching step context. Only a LIVE verdict warrants re-engaging.
  - When a later `codeyam-editor-dev editor advance` prints a
    `> CANCEL STALE WAKEUP: … CancelWakeup <schedule_id>` directive, the awaited
    step has completed — cancel the named fallback wakeup right then so it never
    fires. The `wakeup-check`-on-fire step above stays as the belt-and-suspenders
    fallback for a cancel that gets missed.
