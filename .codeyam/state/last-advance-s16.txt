CONSOLIDATION-FIRST — edit existing scenarios before adding new ones. This is the
LEAD behavior, not an afterthought. The default failure mode is "new feature → new
scenarios," which accumulates dozens of near-identical app scenarios that differ by
one field — noisy, expensive to recapture, hard to review. For every surface this
feature touches, work in this order:

  1. LIST the existing app scenarios first: `codeyam-editor-dev editor scenarios`.
  2. For EACH existing scenario on a touched surface, decide whether the feature
     changes what it should demonstrate. If so, EDIT its seed/mocks and recapture
     IN PLACE — and RENAME it when the feature changes its meaning (e.g.
     `home-grouped` → a name that reflects the new Labels grid). Folding the new
     surface into an existing scenario is the preferred outcome, not a fallback.
  3. Defer creating a NEW scenario to the next step — and only for a display state
     no edited scenario covers. A new STATE is the bar for "new," not a new data
     variation.

The lens for review — the canonical display states. Each distinct state deserves
its own scenario; use this list to judge whether an existing scenario has gone
stale or whether a state is missing entirely:
  - Empty: zero rows / no results / signed-out. Surfaces missing empty-state UI.
  - Single item: one row only. Surfaces "1 result" / single-column collapse.
  - Loaded with many: a rich mix of categories, lengths, ratings, statuses. The canonical demo.
  - All-of-one-kind: every row shares a normally-varying value (all unrated, all draft). Surfaces fallback sort orders.
  - Edge data: very long / short / special-char / missing-optional-field rows. Promote out of "Loaded with many" only when the state has its own visual treatment.

RICH, REALISTIC DATA. "Test Item 1" won't reveal whether the feature works — when
you edit a scenario's seed, use real names, real image URLs (verify-images runs a
HEAD check), realistic copy lengths.

Anti-pattern — "thin parallel scenarios" = SAME display state with cosmetic data
swaps (`Library_Cards1` vs `Library_Cards2`, both populated). DIFFERENT states
(empty vs populated) are NOT thin — they're load-bearing coverage. When enhancing,
enhance WITHIN a state; don't fold empty into populated.
